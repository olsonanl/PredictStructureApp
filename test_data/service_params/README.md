# Service Script Test Params

Params JSON files for testing `App-PredictStructure.pl` end-to-end.
Each file is a complete BV-BRC AppService params payload that can be
passed to the service script.

## Files

| File | Tool | Input mode | MSA | Notes |
|------|------|-----------|-----|-------|
| `esmfold_input_file.json` | ESMFold | input_file | none | Fastest test (~30s) |
| `esmfold_text_input.json` | ESMFold | text_input (inline) | none | Tests inline sequence path |
| `boltz_input_file.json` | Boltz-2 | input_file | none (single-seq) | Requires `msa: empty` converter fix |
| `boltz_msa_upload.json` | Boltz-2 | input_file | upload (.a3m) | Tests MSA file download+use |
| `openfold_input_file.json` | OpenFold 3 | input_file | none | |
| `chai_input_file.json` | Chai-1 | input_file | none | |
| `alphafold_input_file.json` | AlphaFold 2 | input_file | none (uses own pipeline) | Slowest (~26m monomer) |
| `auto_input_file.json` | auto | input_file | none | Tests auto-selection logic |

## Paths

All paths use `/data/...` and `/output` which are standard bind-mount
targets inside the container:

- `/data` <- host `test_data/`
- `/output` <- per-run writable dir

For BV-BRC real-submission testing, replace `input_file`, `output_path`,
and `msa_file` with real workspace paths (`ws:/user@patricbrc/...`).

## Running via the Perl service script

Direct invocation inside the container:

```bash
apptainer exec --nv \
  --bind <test_data>:/data \
  --bind <tmp_out>:/output \
  --bind /local_databases:/local_databases \
  <folding_sif> \
  perl /kb/module/service-scripts/App-PredictStructure.pl \
    http://localhost \
    /kb/module/app_specs/PredictStructure.json \
    /path/to/params.json
```

The `ApptainerRunner.service()` helper in
`tests/acceptance/conftest.py` does exactly this.

## Running via BV-BRC AppService

The BV-BRC test harness submits these params via:

```bash
p3-submit-app PredictStructure params.json
```

or via the service CLI wrapper inside the AppService container.

## Updating the params

When adding new fields to `app_specs/PredictStructure.json`, update the
relevant param files here so test coverage stays in sync. Both the
acceptance tests (tests/acceptance/test_phase3_service_script.py) and
BV-BRC test harness read these files.

## Q/A format for external testing frameworks

For the tier ladder cases (`tier{1-5}_{tool}.json`), each input has a
sibling `tier{N}_{tool}.expected.json` that declares the expected
post-run state in pure JSON (no Python). External testing frameworks
can run the input through `App-PredictStructure.pl` and verify the
predicates without depending on pytest.

### File pairs

| Input | Expected contract |
|---|---|
| `tier1_boltz.json` | `tier1_boltz.expected.json` |
| `tier1_chai.json` | `tier1_chai.expected.json` |
| ... | (one expected per input) |

Both files are auto-generated by `scripts/generate_service_params.py`.
Run the generator with `--check` in CI to catch hand-edits.

### `<case>.expected.json` schema

```jsonc
{
  "name": "tier1_boltz",
  "description": "Boltz on the tier1 fixture",
  "input": {
    "params_file": "tier1_boltz.json",   // sibling input filename
    "needs_gpu": true,
    "needs_workspace_token": false        // if true, runner reads PATRIC_TOKEN_PATH
  },
  "expected": {
    "exit_code": 0,
    "timeout_s": 3600,
    "output_dir": "/output/output",       // container-side dir holding the unified layout
    "files_exist": ["model_1.pdb", "confidence.json", ...],
    "files_forbidden": [],                // optional regression guards
    "schemas": {
      "confidence.json": "schemas/confidence.schema.json",
      "metadata.json":   "schemas/metadata.schema.json",
      "results.json":    "schemas/results.schema.json"
    },
    "json_constraints": {
      "confidence.json": {
        "plddt_mean":           {"min": 0, "max": 100},
        "per_residue_plddt[*]": {"min": 0, "max": 100},
        "per_residue_plddt":    {"length": 46}
      },
      "metadata.json": {
        "tool": {"equals": "boltz"}
      },
      "results.json": {
        "outputs[*].sha256": {"oneOf": [{"matches": "^[0-9a-f]{64}$"}, "null"]}
      }
    },
    "pdb_constraints": {
      "model_1.pdb": {"min_atoms": 40, "min_size_bytes": 1000, "first_record_type": "ATOM"}
    },
    "stdout_contains": [],                // optional log-message check
    "stderr_contains": []
  }
}
```

### Predicate atoms (used inside `json_constraints`)

| Atom | Meaning |
|---|---|
| `{"equals": v}` | strict equality |
| `{"min": x, "max": y}` | numeric range (either bound optional) |
| `{"length": n}` | array/string length |
| `{"matches": "regex"}` | regex on string |
| `{"oneOf": [...]}` | disjunction; passes if any sub-atom matches |
| `"null"` | explicit null |

Field paths support dot-notation and `[*]` for array fan-out:

| Path | Resolves to |
|---|---|
| `plddt_mean` | the top-level scalar |
| `per_residue_plddt` | the array itself (use with `length`) |
| `per_residue_plddt[*]` | every element of the array |
| `outputs[*].sha256` | every `sha256` field across `outputs[]` |

### Reference runner

`scripts/run_qa_case.py <case>.expected.json --sif <path>` is a 200-line
stdlib + jsonschema implementation that the external framework can
invoke directly or re-implement in any language.

```bash
# Run one case
python scripts/run_qa_case.py \
  test_data/service_params/tier1_esmfold.expected.json \
  --sif /scout/containers/folding_260425.1.sif \
  --out output/qa/tier1_esmfold.result.json

# Exit 0 iff all predicates passed; result JSON has the failure list.
```

Result JSON shape:

```jsonc
{
  "name": "tier1_esmfold",
  "passed": true,
  "elapsed_s": 38.4,
  "exit_code": 0,
  "summary": {"checked": 23, "failed": 0},
  "failures": []
}
```

### What the format does NOT cover

- **Workspace upload roundtrip** (`p3-cp` to a real workspace) -- needs a
  PATRIC token and cleanup; out of scope for offline Q/A.
- **CWL workflow execution** -- different runner shape; covered by
  `tests/acceptance/test_cwl_workflow_execution.py` instead.
- **Byte-exact regression against a golden `model_1.pdb`** --
  predictions are non-deterministic across hardware (CUDA kernels,
  fp16); contract checks are more durable than golden bytes.
